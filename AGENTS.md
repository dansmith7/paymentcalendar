# This is NOT the Next.js you know

This version has breaking changes — APIs, conventions, and file structure may all differ from your training data. Read the relevant guide in `node_modules/next/dist/docs/` before writing any code. Heed deprecation notices.

## Базовые принципы

- Если фичу можно сделать в 100 строк вместо 500, агент обязан выбрать 100 строк.
- Перед правками агент сначала читает существующий код и следует текущей архитектуре проекта.
- Агент не добавляет новые абстракции, библиотеки и слои, если задача решается существующими средствами проекта.
- Любая правка должна быть минимальной по зоне воздействия: менять только те файлы, которые нужны для задачи.
- Агент не перетирает незакоммиченные изменения пользователя и сначала проверяет `git status`.
- Перед изменением Next.js-кода агент сверяется с локальной документацией `node_modules/next/dist/docs/`, потому что версия Next.js в проекте нестандартная.
- Бизнес-логика заявок должна оставаться в `lib/services` и `lib/repositories`, а UI не должен напрямую знать детали Supabase.
- Валидация пользовательского ввода должна проходить через существующие Zod-схемы, а не через разрозненные проверки в компонентах.
- Для правок UI агент использует существующий визуальный стиль, Tailwind-паттерны и компоненты проекта.
- После значимых изменений агент запускает минимум `npm run lint` и `npx tsc --noEmit`, если задача затрагивает TypeScript/React.
- Если меняется логика данных Supabase, агент проверяет миграции, типы и репозиторный слой вместе.
- Документация и код не должны расходиться: если поведение изменилось, агент обновляет README/docs или отмечает расхождение.
- Секреты, токены и ключи нельзя вставлять в код, коммиты или ответы. Раскрытый токен считается скомпрометированным.
- Бинарники, локальные инструменты и временные файлы не должны попадать в git.
- Перед тем как сообщать, что всё готово, агент должен открыть проект локально в браузере, проверить изменения, протестировать ключевой сценарий и убедиться, что всё работает. Нельзя приходить к пользователю с сообщением, что всё готово, пока эти пункты не выполнены.
- В dev-режиме `AUTH_MODE=mock` должен использовать отдельную локальную песочницу без Supabase; тестовые записи допустимы только в эту песочницу. Реальные таблицы Supabase нельзя менять из локального браузерного тестирования. В `AUTH_MODE=supabase` + `NODE_ENV=development` записи в Supabase должны быть заблокированы, если явно не включён `ALLOW_DEV_WRITES=true`.
- Данные в Supabase критичны: агент не должен создавать миграции или команды, которые удаляют таблицы, пересоздают таблицы с потерей данных, очищают данные или физически удаляют строки, если пользователь явно не подтвердил конкретное destructive-действие.
- Миграции по умолчанию должны быть additive/safe: `create table if not exists`, `alter table add column if not exists`, индексы, безопасные `update`, новые функции/триггеры. Операции `drop table`, `truncate`, массовый `delete`, пересоздание таблиц и destructive reset запрещены без отдельного явного согласования.
- Перед любыми правками Supabase-схемы агент проверяет существующие миграции и ищет destructive SQL, чтобы повторный push/deploy не мог случайно удалить рабочие таблицы или данные.

## Правила проекта

- Основной поток данных: UI -> server action/page -> `lib/data` -> `lib/services` -> `lib/repositories` -> Supabase.
- Заявки на оплату нельзя физически удалять; сценарии скрытия должны использовать `deleted_at`. Это частный случай общего правила: данные проекта не удаляются физически без явного отдельного подтверждения.
- Роли `employee`, `manager`, `admin` должны проверяться на сервере, а не только в интерфейсе.
- Mock auth используется только для разработки; real auth идет через Supabase magic link.
- Mock auth не должен читать или писать заявки/профили Supabase; dev-заявки живут в `.dev-data/mock-payment-store.json`.
- Excel-экспорт, аналитика и недельные группировки должны использовать существующие helpers, а не дублировать календарную логику.
