# Сохранность данных по заявкам

## Что сделано в проекте

1. **Колонка `deleted_at`** в `payment_requests` — мягкое скрытие. Пока колонки нет в вашей БД, приложение **не** фильтрует по ней (иначе PostgREST падает). Добавьте колонку миграцией `20260413140000_payment_requests_deleted_at_column.sql` (или полной `20260413100000_payment_requests_safety.sql`). Когда колонка есть и вы начнёте заполнять `deleted_at`, снова добавьте в `lib/repositories/requests-repository.ts` на чтение/обновление условие `.is("deleted_at", null)`.

2. **Запрет физического `DELETE`** — триггер `trg_payment_requests_block_delete` на `payment_requests`: любая попытка удалить строку завершится ошибкой с понятным текстом.

3. **Журнал `payment_request_audit`** — после каждого `INSERT` и `UPDATE` по заявке пишется строка с `old_record` / `new_record` (JSON). Просмотр: SQL Editor в Supabase, таблица `payment_request_audit`.

4. **В коде нет вызовов `delete` по заявкам** — при ревью не добавлять удаление без явной политики.

## Что нужно сделать вам

- **Миграции по желанию:** только колонка — `20260413140000_payment_requests_deleted_at_column.sql`; аудит и запрет `DELETE` — `20260413100000_payment_requests_safety.sql`.
- **Резервное копирование** — в панели Supabase включите подходящий тариф с автоматическими бэкапами / PITR.
- **Миграции** — по возможности только добавление колонок и безопасные `UPDATE` + смена `CHECK`; не пересоздавать таблицу с потерей данных.
- **Скрытие заявки** — когда появится сценарий «убрать из работы», выставляйте `deleted_at = now()` через отдельное действие (сейчас UI не добавлялся — только инфраструктура).

## Восстановление заявки

```sql
update public.payment_requests
set deleted_at = null
where id = '<uuid>';
```

Выполнять осознанно (роль с правом на `UPDATE`).
